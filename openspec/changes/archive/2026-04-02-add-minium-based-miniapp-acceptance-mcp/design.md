## Context

`miniprogram-minium-mcp` 当前还是一个实验仓库，只有 OpenSpec 变更、README 与基础目录占位，尚未有实际的 MCP Server、Minium 适配层或测试代码。该变更的目标不是再造一个通用小程序自动化框架，而是把 Minium 已有的运行时驱动能力封装成一组更适合 AI Agent 调用的高层验收工具。

这里有几个显式约束：

- 底层自动化依赖 Minium 和本地微信开发者工具运行环境，意味着服务端必须处理本地依赖存在、启动失败、连接中断等非纯代码问题。
- 上层调用方是 AI Agent，而不是熟悉 Minium API 的人工测试开发，因此工具接口必须偏“任务导向”和“结构化返回”，不能把底层对象模型直接暴露出去。
- 仓库当前没有既有实现包袱，因此可以优先围绕“最小可行验收链路”设计，但也需要为后续扩展到更多动作、断言和报告格式预留分层。
- 该 MCP 的定位是“小程序验收与测试”，不是任意调试代理，因此需要在设计阶段就收紧能力边界，避免出现可任意执行底层命令、篡改运行状态或泄漏过多环境信息的工具面。

主要相关方包括：

- 使用 Codex、ChatGPT 或其他 Agent 的开发者/测试同学，他们希望通过工具化接口执行验收流程。
- 本地执行环境维护者，他们需要能理解依赖准备、失败原因和产物落点。
- 后续实现者，他们需要清晰的服务分层和错误模型，避免在代码阶段重新讨论接口形态。

## Goals / Non-Goals

**Goals:**

- 提供一个基于 Python + Minium 的 MCP Server，支持 AI Agent 以工具调用方式完成小程序验收主链路。
- 提供显式的会话管理，对外统一支持“提供项目路径 -> 服务自动完成开发者工具拉起、auto 和 attach -> 维持会话 -> 获取上下文 -> 结束会话”的稳定交互模型。
- 提供高层页面动作与断言能力，覆盖最小可行验收集：当前页面读取、元素查询、点击、输入、等待、断言、截图。
- 为失败场景提供结构化取证能力，让 Agent 在断言失败或动作超时时拿到可读的错误原因、截图路径和上下文摘要。
- 在工具层明确安全边界，只暴露验收所需能力，不直接透传 Minium 原生对象或任意驱动操作。

**Non-Goals:**

- 不在首期实现完整的 Minium API 映射，也不支持任意脚本执行、任意 JS 注入或任意运行态修改。
- 不在首期覆盖复杂调试能力，例如网络代理、性能剖析、全量日志采样、组件树深度诊断。
- 不把该服务设计成远程共享的多租户云服务，首期默认面向本地单用户开发/测试环境。
- 不在本次设计中解决 CI 中如何安装和驱动微信开发者工具的全部问题，只为后续本地优先的自动化留出迁移路径。

## Decisions

### 1. 使用“`MCP Server -> 验收服务层 -> Minium 适配层 -> 产物/环境层`”的四层结构

设计：

- `src/server/`：MCP Server 入口，负责工具注册、参数校验、返回结构封装。
- `src/domain/`：验收服务层，封装会话生命周期、页面动作、断言、取证逻辑。
- `src/adapters/minium/`：对 Minium 做薄适配，统一底层调用、异常翻译和对象转换。
- `src/support/`：产物目录、时间戳、配置加载、日志、截图路径管理等横切能力。

原因：

- MCP 协议处理与 Minium 驱动逻辑关注点不同，分层后可以避免工具函数直接堆满底层调用细节。
- 后续若底层驱动从 Minium 扩展到其他方案，只需替换适配层，不必重写工具契约。
- 当前仓库为空，实现初期就建立分层成本很低，但能显著降低后续扩展耦合。

备选方案：

- 方案 A：所有逻辑直接写在 MCP tool handler 中。
  - 优点：启动快，POC 成本最低。
  - 缺点：会话、断言、截图、错误翻译会高度重复，后续几乎必然重构。
- 方案 B：先只写单层服务，后续再拆。
  - 优点：实现简单。
  - 缺点：容易把底层 Minium 语义直接带进工具层，不利于形成稳定的 Agent 接口。

结论：采用四层结构，优先把“协议层”和“驱动层”隔离开。

### 2. 会话采用“显式创建、短期保活、单会话串行操作”的状态模型

设计：

- 提供独立的会话创建工具，对外以项目路径作为主输入，由服务内部完成拉起和附着流程，并返回 `session_id`。
- 后续动作类工具必须显式传入 `session_id`，服务端按会话查找底层 Minium 实例。
- 每个会话维护最近一次页面路径、最近截图、最近错误和最后活跃时间。
- 首期默认单进程内存态会话仓库，不做跨进程持久化；设置空闲超时，超时后自动清理。
- 首期对同一 `session_id` 上的动作按串行执行处理，避免多个 Agent 或多个 tool call 并发操作同一 Minium 实例引发竞态。
- 对外不暴露多目标选择、目标名称或目标端口等会话选择参数，避免把底层 DevTools 连接细节泄漏到工具协议层。

原因：

- AI Agent 天生是多轮工具调用模式，若不显式建模会话，后续每个动作都要重新连接环境，稳定性差且上下文无法继承。
- Minium 驱动对象通常不适合跨请求重新序列化，内存态会话管理是最直接且可控的落地方式。
- 串行化虽然牺牲吞吐，但显著降低首期并发竞态复杂度，更贴合本地验收场景。

备选方案：

- 方案 A：无状态工具，每次动作都重新启动或重新连接。
  - 优点：实现简单，服务端几乎无状态。
  - 缺点：成本高、失败率高，且无法保留验收上下文。
- 方案 B：做持久化会话仓库。
  - 优点：更接近长生命周期服务。
  - 缺点：当前没有真实需求，且会引入序列化、恢复和锁管理复杂度。

结论：首期采用显式内存会话模型，并通过超时清理和串行访问保证稳定性。

### 3. 工具接口采用“高层任务语义 + 结构化结果”，而不是暴露底层元素对象

设计：

- 工具面聚焦以下能力：
  - 会话类：创建会话、关闭会话、获取当前页面、截图、读取最近日志/错误摘要。
  - 动作类：查询元素、点击元素、输入文本、等待条件成立。
  - 断言类：断言页面路径、断言元素文本、断言可见性/存在性。
- 元素定位统一用结构化定位器描述，而不是直接返回 Minium 元素句柄。
- 查询结果返回可序列化摘要，例如元素数量、首个元素文本、是否可见、用于后续动作的定位参数回显。
- 断言工具失败时返回统一错误码、断言描述、实际观测值和证据路径。

原因：

- MCP 工具返回必须天然可序列化；底层元素对象若直接暴露，会让后续调用无法稳定复用。
- AI Agent 更需要“我能不能点击登录按钮”“当前页面是不是订单页”这类高层语义，而不是自己拼接底层调用。
- 高层接口更容易做参数校验和边界收敛，也更符合“验收型 MCP”定位。

备选方案：

- 方案 A：直接镜像 Minium API。
  - 优点：开发快，功能覆盖广。
  - 缺点：学习成本高、错误模型分散、暴露能力过强。
- 方案 B：完全只提供场景化脚本工具，例如“登录”“下单”。
  - 优点：Agent 调用最简单。
  - 缺点：过度绑定单一业务，不适合作为通用验收基础设施。

结论：采用高层但通用的任务语义接口，位于底层驱动与业务流程脚本之间。

### 4. 定位器采用“首期统一 selector 描述，后续可扩展多策略”

设计：

- 首期统一接受结构化定位器对象，例如：
  - `type`: `css`、`text`、`id`
  - `value`: 对应定位值
  - `index`: 可选，用于多元素场景
- 服务层把定位器转换为 Minium 可用查询方式，并统一处理“未找到”“找到多个”“元素不可交互”等常见问题。
- 不在首期暴露复杂组合定位 DSL，避免参数面过大。

原因：

- 纯字符串 selector 虽然简单，但难以向 Agent 明确表达不同定位策略。
- 结构化定位器更适合 schema 校验，也方便未来增加 `role`、`data-testid`、`path` 等策略。

备选方案：

- 方案 A：只接受原始字符串 selector。
  - 优点：最简单。
  - 缺点：歧义大，扩展困难。
- 方案 B：设计完整 DSL。
  - 优点：表达力强。
  - 缺点：首期复杂度过高，不利于 Agent 正确构造参数。

结论：首期使用轻量结构化定位器，在“可表达”与“易使用”之间取平衡。

### 5. 失败处理采用统一验收错误模型，并自动附带证据产物

设计：

- 定义统一错误分类：
  - `ENVIRONMENT_ERROR`：依赖缺失、开发者工具未启动、连接失败。
  - `SESSION_ERROR`：会话不存在、会话失活、会话状态异常。
  - `ACTION_ERROR`：点击/输入/等待动作失败。
  - `ASSERTION_FAILED`：断言不成立。
  - `INTERNAL_ERROR`：服务内部未预期异常。
- 对动作失败和断言失败，服务层自动触发一次取证：
  - 截图保存到 `artifacts/` 下按会话分目录存放。
  - 记录页面路径、定位器摘要、超时参数、异常堆栈摘要。
  - 返回结构中带上 `artifact_paths` 与 `debug_context`。
- MCP 返回面向 Agent 保持简洁，但保留 `error_code`、`message`、`details`、`artifacts` 四类字段。

原因：

- 验收场景下最重要的不是“抛异常”，而是“失败后还能继续判断和修复”。
- 统一错误模型可以避免底层异常直接泄漏给 Agent，减少噪音并提高可恢复性。

备选方案：

- 方案 A：底层异常原样透传。
  - 优点：开发最省事。
  - 缺点：信息不稳定，不适合 Agent 解析。
- 方案 B：只返回一句错误描述，不落截图。
  - 优点：实现简单。
  - 缺点：验收失败时缺乏足够证据，排障价值很低。

结论：统一错误模型 + 自动取证是该 MCP 能否被实际使用的核心设计之一。

### 6. 首期以本地单用户运行模型为主，配置通过环境变量和轻量配置文件注入

设计：

- 通过环境变量或本地配置文件提供 Minium 运行所需参数，例如项目路径、开发者工具路径、超时默认值和固定自动化端口。
- 服务启动时进行配置校验，但不强制所有字段都预检；与会话创建强相关的依赖在创建时再做二次校验。
- 默认所有产物落盘到仓库 `artifacts/` 目录，便于与仓库上下文一起查看。

原因：

- 当前目标用户是本地开发者，复杂的远程配置中心或账户体系没有必要。
- 环境变量 + 配置文件的组合最适合本地实验和后续 CI 迁移。

备选方案：

- 方案 A：所有参数都在每次 tool call 里传入。
  - 优点：完全显式。
  - 缺点：调用冗长，且不利于多轮验收复用。
- 方案 B：只用硬编码默认值。
  - 优点：最省事。
  - 缺点：几乎不可迁移，也不利于多人使用。

结论：采用本地配置驱动，兼顾实验效率与后续可迁移性。

### 7. 所有面向调用方的输出采用中英文双语支持，并按环境语言自动选择

设计：

- 服务启动时读取本地语言环境或显式语言配置，默认把中文环境识别为中文输出，其余环境统一降级为英文输出。
- 所有面向调用方的文本输出，包括工具成功信息、错误消息、CLI 帮助和调试摘要，统一通过消息模板层生成，而不是在业务代码中散落硬编码文案。
- 结构化返回保持字段名稳定，语言切换只作用于可读消息内容，避免影响工具协议兼容性。

原因：

- 该 MCP 面向 AI Agent 和本地开发者使用，运行环境可能不是中文系统，直接固定中文输出会降低可读性和跨团队可用性。
- 语言选择如果晚于工具实现再补，后续会导致成功消息、错误文案和文档样例大面积返工。
- 保持字段名不变、仅切换消息内容，可以兼顾人类可读性和工具集成稳定性。

备选方案：

- 方案 A：始终输出中文。
  - 优点：实现最简单。
  - 缺点：不满足非中文环境使用场景。
- 方案 B：由每个工具单独决定输出语言。
  - 优点：局部灵活。
  - 缺点：会导致文案风格和语言判断规则不一致。

结论：采用统一语言检测与消息模板层，中文环境输出中文，非中文环境输出英文。

## Risks / Trade-offs

- [Minium 与微信开发者工具环境耦合较强] → 在会话创建前补充环境探测与错误提示，优先把“缺什么、怎么补”说清楚，而不是只报连接失败。
- [内存态会话在服务重启后会丢失] → 首期接受该限制，并要求 Agent 在长流程开始时显式创建会话；后续若出现真实需求，再补持久化恢复。
- [单会话串行执行会限制并发] → 首期优先稳定性；若后续出现并发验收需求，再基于会话级锁或多进程模型扩展。
- [统一高层工具可能覆盖不全 Minium 特性] → 在 spec 中先覆盖最小可行链路，保留适配层扩展点，而不是一开始追求底层能力全量开放。
- [截图和日志产物可能快速膨胀] → 采用按会话分目录、按时间戳命名，并在会话关闭或周期性清理时支持保留最近 N 份产物。
- [AI Agent 可能构造不稳定定位器] → 通过结构化定位参数约束输入，并在查询结果中返回摘要，帮助 Agent 调整下一步动作。

## Migration Plan

1. 先建立服务骨架和目录结构，完成基础配置加载、MCP Server 启动和会话仓库。
2. 实现 Minium 适配层最小能力：基于项目路径拉起开发者工具、执行 `auto + attach`、获取当前页面、查询元素、截图。
3. 在服务层补齐动作与断言封装，并统一错误模型与证据产物输出。
4. 基于 OpenSpec 里的 capability specs 收敛工具契约，再补 `tasks.md` 细化实现顺序。
5. 添加最小集成验证，覆盖“创建会话 -> 获取当前页面 -> 查询元素 -> 截图 -> 断言”的 happy path 和至少一个失败路径。

回滚策略：

- 该变更为新增能力，不涉及线上替换；若实验实现不可用，可以通过不发布该 MCP Server 或移除新增入口文件回退。
- 若已经生成局部代码但能力不稳定，可仅保留适配层与产物目录约定，暂不开放对外工具注册，避免暴露半成品接口。

## Open Questions

- 当前目标环境下是否需要保留纯 `attach` 作为内部兼容路径，还是长期统一收敛为“项目路径驱动的 `auto + attach`”模式？
- 首期 MCP 工具协议是直接面向通用 MCP schema，还是需要在返回中额外约定更细的 `debug_context` 字段结构？
- 元素定位首期是否需要支持基于文案的模糊匹配，还是只支持精确匹配以提升稳定性？
- 是否要在首期就支持页面级日志抓取，还是仅在失败时返回错误摘要和截图即可？
- 本地验证是否需要提供一个示例小程序工程，还是只提供接入说明和最小 smoke test 脚本？
