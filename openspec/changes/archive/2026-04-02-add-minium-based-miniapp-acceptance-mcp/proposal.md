## Why

当前微信小程序自动化生态更偏向人工调试脚本或传统测试框架，缺少一套直接面向 AI Agent 的验收接口。我们希望基于 Minium 构建一个类似 Playwright 使用体验的 MCP Server，让 Agent 能以结构化工具方式启动小程序、读取页面、执行交互、做断言并输出验收证据，从而把“小程序手工验收”升级为“可编排的 AI 验收流程”。

## What Changes

- 新增一个基于 Minium 的小程序验收型 MCP Server，面向 AI Agent 暴露高层测试与验收工具。
- 新增会话管理能力，对外统一采用“提供项目路径，由服务自动完成开发者工具拉起、auto 和 attach”的模式，并在一次验收流程中维持稳定会话。
- 新增页面与元素交互能力，支持获取当前页面、查询元素、点击、输入、等待条件成立等常见验收动作。
- 新增断言与取证能力，支持对页面路径、元素文本、可见性等进行结构化断言，并在失败时输出截图、日志或上下文信息。
- 明确该 MCP 的边界是“验收与测试”，而不是任意调试和任意修改运行时状态，避免 AI 直接获得过强的底层驱动能力。

## Capabilities

### New Capabilities
- `miniapp-acceptance-session`: 面向 AI 的小程序验收会话管理能力，覆盖项目拉起、自动附着、当前页面获取、日志与截图取证、异常上下文输出。
- `miniapp-acceptance-actions`: 面向 AI 的页面交互与断言能力，覆盖导航、元素查询、点击、输入、等待、断言和验收失败反馈。

### Modified Capabilities
- 无

## Impact

- 新增一个独立仓库内的 MCP Server 实现，预计以 Python + Minium 为主，必要时补充轻量启动脚本或适配层。
- 会引入 Minium 及其运行所需依赖，并要求本地存在可用于自动化的小程序运行环境与微信开发者工具。
- 需要设计一套适合 AI 使用的高层工具协议与错误模型，而不是直接透传 Minium 的底层接口。
- 需要补充最小可行的集成验证路径，至少覆盖“提供项目路径 -> auto + attach -> 获取当前页面 -> 查询元素 -> 截图/断言”主链路。
