## Context

当前仓库里的验收型 MCP 已经形成了比较清晰的分层：MCP tool 层暴露会话与动作接口，领域服务层负责会话状态、断言和失败取证，Minium 适配层负责把高层语义翻译为底层运行时调用。现有能力覆盖点击、输入、等待、断言和截图，适合单步验收动作，但还不能表达“先按下且保持、随后移动、再由另一指插入点击”的连续复合手势。

这次变更的特殊点不在于“多几个动作 API”，而在于交互语义从“单次动作立即完成”扩展为“跨多次 tool call 维持一组活跃触点”。这会同时影响：

- 会话模型：需要在会话里维护活跃触点和最近一次手势事件摘要。
- MCP 协议：需要在保持高层验收语义的前提下暴露最小触点原语。
- 运行时适配：需要根据当前活跃触点集合构造 `touches` 与 `changedTouches`，而不是只触发单个元素的点击。
- 失败取证：截图之外，还需要返回触点状态和本次事件摘要，否则难以调试。

同时，这个仓库已经在真实运行时里引入了基于 touch 事件的点击回退逻辑，说明底层 Minium 事件能力是可用的；但设计层面仍然要坚持“验收型 MCP”边界，避免演化成通用脚本执行器。

主要相关方包括：

- 通过 Agent 编排小程序验收流程的开发和测试同学，他们需要可组合但稳定的手势原语。
- 负责维护 MCP 稳定性的实现者，他们需要一套不会把底层 Minium 细节无限外泄的协议。
- 需要排查复杂交互失败原因的使用者，他们依赖结构化取证和上下文摘要，而不仅是截图。

## Goals / Non-Goals

**Goals:**

- 为验收型 MCP 增加最小可行的多触点基础能力，支持单指按住、持续移动、第二指插入点击等复合交互。
- 在会话上下文中显式维护活跃触点状态，使多次独立 tool call 可以组合成连续手势。
- 保持协议高层化，让 Agent 传 `pointer_id`、目标和位置，而不是直接传底层 Minium 对象或原始脚本。
- 为手势失败补充结构化调试信息，包括活跃触点摘要、变更触点、目标解析结果和截图路径。
- 将首期范围收敛在最多两指的基础触点原语上，优先验证典型业务场景而非追求完整手势 DSL。

**Non-Goals:**

- 不在首期提供任意事件脚本回放能力，不支持调用方直接提交原始 `touches` / `changedTouches` 事件数组。
- 不在首期支持旋转、捏合、缩放等命名高级手势模板，除非它们能由基础原语在上层编排得到。
- 不把现有点击、输入等动作全部重构为新的通用手势引擎入口；首期只新增手势能力并与现有 action 并存。
- 不在本次设计中引入跨会话并发编排、手势录制回放文件格式或远程多租户执行模型。

## Decisions

### 1. 新增独立 capability：`miniapp-acceptance-gestures`

设计：

- 新增 `miniapp-acceptance-gestures` spec，专门定义手势相关的基础能力和失败取证。
- 保持 `miniapp-acceptance-actions` 继续承载点击、输入、等待、断言等通用动作，不把现有 action spec 混成“所有输入事件的大集合”。
- `miniapp-acceptance-session` 只承载会话状态变化和上下文保留相关 requirement，不承载具体手势语义。

原因：

- 手势能力的核心复杂度在于“跨多次调用维持状态”，与现有单步 action 的心智模型明显不同，单独建 capability 更容易保持边界清晰。
- 这样可以让后续 `specs` 阶段分别描述“手势做什么”和“会话如何保留上下文”，避免 requirement 交叉污染。
- 对实现层也更友好：工具注册、服务层和运行时扩展可以独立演进，不必一次性重写现有 action 语义。

备选方案：

- 方案 A：直接扩展 `miniapp-acceptance-actions`，把手势原语也作为 action 的一部分。
  - 优点：spec 数量少，表面上更集中。
  - 缺点：会把“单步动作”和“持续触点状态”混在一个 capability 里，导致 requirement 颗粒度不一致。
- 方案 B：只修改 `miniapp-acceptance-session`，把手势当作会话级低层能力。
  - 优点：能复用会话状态描述。
  - 缺点：会让 session spec 同时负责生命周期和业务动作，职责过重。

结论：新增独立 `miniapp-acceptance-gestures` capability，并只修改 `miniapp-acceptance-session` 以支持相关上下文。

### 2. 会话内维护“活跃触点状态机”，而不是把每个手势调用做成无状态动作

设计：

- 在 session 模型中新增 `active_pointers`，按 `pointer_id` 保存当前仍处于按下状态的触点。
- 每个活跃触点至少包含：
  - `pointer_id`
  - `status`，如 `down`
  - `current_position`
  - `origin_target_summary`
  - `started_at`
- 额外保留 `latest_gesture_event` 摘要，记录最近一次触点动作类型、目标解析结果和触点集合快照。
- 会话清理、关闭和超时回收时，统一丢弃 `active_pointers`，避免出现“会话失效但触点状态残留”的不一致。

原因：

- 你的目标场景本质上依赖“第一根手指仍然按着”的时间重叠语义，如果没有会话级状态，就无法在后续 tool call 中表达这一点。
- 无状态设计会迫使每个调用都重复提交完整的触点集合，这既难让 Agent 正确构造，也会泄漏过多底层事件细节。
- 把触点状态放进 session 后，MCP 只需要表达“这次哪个 pointer 发生了什么变化”，其余活跃触点由服务端自动补全。

备选方案：

- 方案 A：无状态手势调用，每次都由调用方提交完整 `touches` 和 `changedTouches`。
  - 优点：实现直接，对服务端状态要求低。
  - 缺点：调用复杂度高，容易构造错误，也违背高层验收语义。
- 方案 B：仅在运行时层内隐式保存触点，不进入 session 模型。
  - 优点：对领域模型改动较小。
  - 缺点：MCP 返回无法解释当前活跃触点，也不利于失败取证和会话恢复判断。

结论：活跃触点状态必须进入 session 领域模型，并作为公开上下文的一部分被维护和回传。

### 3. 对外暴露“最小触点原语”，不暴露原始事件 DSL

设计：

- 首期只新增四个手势原语：
  - `touch_start`
  - `touch_move`
  - `touch_end`
  - `touch_tap`
- 每个原语都要求 `session_id` 和 `pointer_id`，目标参数支持：
  - `locator`
  - 绝对页面坐标 `x / y`
- `touch_move` 可选支持 `duration_ms` 或 `steps`，用于表达平滑移动；首期不要求调用方手工提交中间事件列表。
- 服务端负责把高层目标解析为页面坐标，并按当前 `active_pointers` 组装底层 `touches` / `changedTouches`。

原因：

- 对 Agent 来说，最自然的表达不是“构造一份复杂事件脚本”，而是“pointer 1 开始按住”“pointer 1 移动到这里”“pointer 2 点一下按钮 B”。
- 仅开放最小原语可以保留足够表达力，同时避免通用 DSL 把能力边界扩大为任意事件注入。
- 目标同时支持 `locator` 和 `x / y`，可以兼顾“按住某个控件”和“沿光标路径拖动到特定位置”两类典型场景。

备选方案：

- 方案 A：直接开放 `dispatch_touch_sequence(events[])`。
  - 优点：表达力最强。
  - 缺点：协议复杂、调试困难，也容易越过“验收型 MCP”边界。
- 方案 B：只开放命名高级手势，如 `pinch`、`rotate`、`two_finger_drag`。
  - 优点：调用更简单。
  - 缺点：无法覆盖大量业务定制场景，也不利于 Agent 组合自己的验收流程。

结论：采用最小触点原语模型，由服务端承担状态组合和底层事件拼装责任。

### 4. 首期明确限制为“两指以内”，优先支持组合验证而非通用手势引擎

设计：

- 首期保证最多两个活跃 `pointer_id` 同时存在。
- requirement 和实现都围绕以下能力展开：
  - 单指按下保持
  - 单指持续移动
  - 第二指在第一指未松开时点击
  - 触点释放后的状态回收
- 如果调用方尝试创建超过两个活跃触点，服务返回结构化错误，说明超出当前能力边界。

原因：

- 当前已知业务诉求聚焦“两指重叠交互”，直接支持两指就足够覆盖最重要的验证路径。
- 限定两指可以显著降低 session 状态、调试输出和测试覆盖的复杂度。
- 这也让未来扩展更可控：如果后续确实有三指需求，再基于真实用例扩容，而不是现在提前设计一个通用手势引擎。

备选方案：

- 方案 A：从首期就支持任意数量的 `pointer_id`。
  - 优点：理论上更通用。
  - 缺点：实现、取证和测试复杂度明显上升，当前没有足够需求支撑。
- 方案 B：只支持单指按住与移动，不支持第二指插入。
  - 优点：实现更简单。
  - 缺点：无法覆盖这次 change 的核心业务场景。

结论：首期能力边界明确收敛在“两指以内的基础复合交互”。

### 5. 失败取证除截图外，必须返回触点上下文摘要

设计：

- 所有手势失败统一继续复用当前截图取证机制。
- 同时在错误详情中补充：
  - `event_type`
  - `pointer_id`
  - `active_pointers`
  - `resolved_target`
  - `current_page_path`
  - 可选的 `touches_summary`
- 成功响应也返回精简版 `active_pointers_summary`，便于 Agent 决定下一步是否继续移动、插入第二指或释放触点。

原因：

- 手势问题往往不是“有没有点到”，而是“当时还有哪些手指处于按下状态、坐标是多少、服务端最终把目标解析成了什么位置”。
- 仅靠截图很难判断失败是由定位失败、坐标漂移还是触点状态丢失导致。
- 把触点摘要纳入标准返回也能帮助 Agent 调整下一步策略，而不是盲试。

备选方案：

- 方案 A：仅沿用现有截图取证。
  - 优点：复用现有机制，改动少。
  - 缺点：对复杂手势排障帮助有限。
- 方案 B：把完整底层事件日志全部返回。
  - 优点：信息最全。
  - 缺点：输出噪音大，且会暴露过多底层实现细节。

结论：返回结构化触点摘要，而不是只给截图或直接透传底层事件日志。

## Risks / Trade-offs

- [触点状态和真实运行时不同步] → 每次手势调用后都以服务端 session 作为单一事实来源，并在关闭/超时回收时统一清理活跃触点。
- [坐标解析不稳定导致手势偏移] → 首期只支持 `locator` 与绝对页面坐标两种目标形式，并把解析后的目标坐标写入返回与错误详情中。
- [不同小程序组件对合成 touch 事件兼容性不同] → 先把 requirement 聚焦在基础触点原语，不承诺所有高级业务控件都能 100% 等价于真机高级手势。
- [两指限制可能无法覆盖少量复杂场景] → 先满足已知核心场景，后续若出现真实三指需求，再基于新 change 扩展边界。
- [新增手势工具后与现有 click 边界模糊] → 保持 click 继续代表“一次性点击动作”，而 `touch_*` 明确用于需要跨调用持有触点状态的场景。

## Migration Plan

1. 在 proposal 基础上补齐两个 spec 方向：
   - 新建 `miniapp-acceptance-gestures`
   - 修改 `miniapp-acceptance-session`
2. 在领域模型中定义 `active_pointers` 和 `latest_gesture_event` 的会话状态结构。
3. 在运行时适配层设计目标解析与触点集合拼装逻辑，把高层 `touch_*` 原语映射到底层 Minium touch 事件。
4. 在 MCP tool 层新增手势工具注册和统一响应结构，并让失败路径复用现有截图取证同时补充触点摘要。
5. 为占位运行时和真实运行时分别补测试，覆盖单指保持、多步移动、第二指点击、会话关闭清理和失败取证路径。

回滚策略：

- 该变更属于新增 capability，不影响现有 `click`、`input`、`wait`、`assert` 工具的既有契约。
- 如果实现中发现真实运行时兼容性不足，可以先保留 spec 和设计，推迟工具对外开放，或者仅保留单指保持与移动能力作为过渡版本。

## Open Questions

- `touch_move` 首期是否需要支持 `steps` 与 `duration_ms` 两种平滑控制参数，还是只保留一种以收敛协议？
- 在真实运行时里，目标坐标应统一按页面绝对坐标返回，还是需要同时返回相对元素坐标以帮助调试？
- 是否需要首期提供独立的只读工具查看当前 `active_pointers`，还是把该信息始终附带在成功/失败响应中就足够？
- 占位运行时对多触点场景要模拟到什么粒度，才能既支持测试又不把测试桩做成第二套手势引擎？
